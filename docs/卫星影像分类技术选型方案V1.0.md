## 2. 项目技术需求概述

### 2.1 任务特征

| 特征 | 描述 | 对选型的影响 |
|------|------|-------------|
| 任务类型 | 图像分类（10 类土地覆盖） | 需要分类模型 + CrossEntropyLoss |
| 图像尺寸 | 64×64 像素（极小） | 需上采样至 224×224 匹配预训练模型 |
| 数据规模 | 27,000 张（各类均衡） | 中等规模，迁移学习优于从头训练 |
| 类别均衡性 | 各类约 2,700 张，基本均衡 | 无需类别加权或 focal loss |
| 场景复杂度 | 大尺度土地覆盖，类间有一定混淆 | 需混淆矩阵分析识别易混淆类别对 |

### 2.2 硬件约束

| 约束项 | 具体指标 | 对选型的约束 |
|--------|---------|-------------|
| GPU 显存 | 4 GB GDDR5 | 模型参数量 ≤ 30M（推荐 ≤ 12M）；batch=256 时显存 ≤ 3.0 GB |
| CUDA 架构 | Pascal SM 6.1 | 无硬件 FP16 Tensor Core，FP16 量化无加速收益 |
| CPU 内存 | ≤ 8 GB | num_workers ≤ 4 |
| Python 版本 | 3.8.20（严格） | 所有依赖包必须兼容 Python 3.8 |
| 操作系统 | Windows 10 64-bit 主环境 | 避免依赖 Linux 专属特性（如 `os.fork`） |

### 2.3 关键技术挑战

| 挑战 | 难度 | 应对策略 |
|------|------|---------|
| 64×64 极小图像上采样至 224×224 | 低 | 双线性插值 + ImageNet 标准化，标准操作 |
| 小图信息量有限，模型可能过拟合 | 中 | 数据增强（翻转+旋转+色彩抖动）+ 早停 + Dropout（预留） |
| 10 类中部分类别光谱相似（如 Industrial vs Residential） | 中 | 混淆矩阵分析 + 必要时解冻骨干微调 |
| 4 GB 显存下最大化 batch_size | 低 | 图像小，batch=256 显存仅 1.5~2.0 GB，余量充足 |

## 3. 深度学习框架选型

### 3.1 候选框架对比

| 维度 | PyTorch | TensorFlow 2.x | PaddlePaddle | MXNet |
|------|---------|---------------|-------------|-------|
| 当前版本（2026） | 2.x（1.12.1 稳定） | 2.x | 2.x | 1.x（维护模式） |
| 动态图支持 | ✅ 原生，Pythonic | ✅ Eager mode（2.x） | ✅ 动态图 | ⚠️ 有限 |
| 预训练模型生态 | ✅ TorchVision（最丰富） | ✅ TF Hub / Keras | ⚠️ 有限 | ❌ 基本停滞 |
| 社区与文档 | ✅ 学术界主流、文档极好 | ✅ 工业界较多 | ⚠️ 以中文社区为主 | ❌ 社区萎缩 |
| 调试体验 | ✅ Python 原生调试 | ⚠️ 静态图残留概念 | ⚠️ 一般 | ❌ 较差 |
| Windows 支持 | ✅ 良好 | ⚠️ 历史上问题多 | ✅ 支持 | ❌ 差 |
| CUDA 11.3 兼容 | ✅ 官方支持 | ⚠️ 需特定版本 | ⚠️ 需验证 | ❌ 不支持 |
| 学习曲线 | ✅ 低（Pythonic） | ⚠️ 中（Keras 易，底层复杂） | ⚠️ 中 | ❌ 高 |

### 3.2 选型结论

**选择：PyTorch 1.12.1 + CUDA 11.3**

**选型理由**：

1. **预训练模型生态最丰富**：TorchVision 提供包括 ResNet18/34/50、MobileNetV3、EfficientNet、ConvNeXt 在内的 80+ 预训练模型，全部支持 `weights='IMAGENET1K_V1'` 一行加载
2. **学术界标准**：EuroSAT 原始论文及后续基准研究几乎全部使用 PyTorch，选 PyTorch 便于与文献结果对比
3. **版本锁定成熟**：PyTorch 1.12.1 + CUDA 11.3 + Python 3.8.20 组合已经过项目验证，兼容 GTX 1050 Ti（SM 6.1）
4. **调试便捷**：动态图 + Python 原生异常处理，对算法研发迭代友好
5. **Windows 原生支持**：主开发环境 Windows 10，PyTorch 在 Windows 上的安装和运行体验显著优于 TensorFlow

**否决 TensorFlow 的理由**：
- TF 2.x 虽支持 Eager mode，但预训练模型生态分散（TF Hub、Keras Applications 两套体系）
- 历史上 Windows + CUDA 适配问题多，需精确匹配 cuDNN 版本
- 本项目不需 TF Serving / TF Lite 等工业部署特性，PyTorch 更轻量

**版本锁定说明**：

| 组件 | 版本 | 锁定理由 |
|------|------|---------|
| Python | 3.8.20 | PyTorch 1.12.1 官方支持的最高 3.8.x 版本 |
| PyTorch | 1.12.1 | 稳定、文档齐全、与 CUDA 11.3 配合成熟 |
| CUDA | 11.3 | Pascal SM 6.1 支持的成熟 CUDA 版本 |
| cuDNN | 8.2+ | CUDA 11.3 配套版本 |
| TorchVision | 0.13.1 | 与 PyTorch 1.12.1 同步发布 |

## 4. 模型架构选型

### 4.1 候选空间

基于需求规格说明书 §2.4 的完整分析，14 种候选模型按精度-效率维度分布如下：

**高精度组（Top-1 > 98%，参数量 21~29M）**：

| 模型 | 参数量 | FLOPs | Acc. | 4 GB 适配 |
|------|--------|-------|------|----------|
| ResNeXt-50 | 25.0M | 4.3G | 98.96% | ⚠️ 可用 |
| EfficientNetV2-S | 21.5M | 2.9G | 98.88% | ⚠️ 可用 |
| DenseNet-161 | 28.7M | 7.8G | 98.83% | ⚠️ 可用 |
| ResNet50 | 25.6M | 4.1G | 98.75% | ⚠️ 可用 |
| ConvNeXt-Tiny | 28.6M | 4.5G | 98.74% | ⚠️ 可用 |

**效率组（Top-1 93~99%，参数量 3~12M）**：

| 模型 | 参数量 | FLOPs | Acc. | 4 GB 适配 |
|------|--------|-------|------|----------|
| MobileNetV3-Large | 5.4M | 0.22G | 98.72% | ✅ 优秀 |
| DenseNet-121 | 8.0M | 2.9G | ~97.5% | ✅ 良好 |
| ResNet34 | 21.8M | 3.7G | ~95.5% | ✅ 良好 |
| EfficientNet-B1 | 7.8M | 0.70G | ~95.0% | ✅ 良好 |
| EfficientNet-B0 | 5.3M | 0.39G | ~94.0% | ✅ 优秀 |
| MobileNetV2 | 3.5M | 0.32G | ~93.5% | ✅ 优秀 |
| ResNet18 | 11.7M | 1.8G | ~93.0% | ✅ 优秀 |

### 4.2 选型决策矩阵

对效率组候选模型进行加权评分（权重：硬件适配 30%、精度 25%、成熟度 20%、训练效率 15%、升级灵活性 10%）：

| 评估维度 | 权重 | ResNet18 | MobileNetV3-L | DenseNet-121 | EfficientNet-B1 | MobileNetV2 |
|---------|------|----------|--------------|-------------|----------------|------------|
| 硬件适配性（4 GB） | 30% | ★★★★★ | ★★★★★ | ★★★★ | ★★★★★ | ★★★★★ |
| EuroSAT 精度 | 25% | ★★★ (93%) | ★★★★★ (98.7%) | ★★★★ (97.5%) | ★★★★ (95%) | ★★★ (93.5%) |
| 成熟度与社区 | 20% | ★★★★★ | ★★★★ | ★★★★ | ★★★ | ★★★★ |
| 训练效率（epoch 耗时） | 15% | ★★★★★ | ★★★★★ | ★★★★ | ★★★★ | ★★★★★ |
| 升级灵活性 | 10% | ★★★★★ | ★★★ | ★★★ | ★★★ | ★★★ |
| **加权综合** | **100%** | **★★★★½** | **★★★★½** | **★★★★** | **★★★★** | **★★★½** |

### 4.3 最终选型结论

**主选：ResNet18**

| 选型因素 | 论证 |
|---------|------|
| **经典可靠性** | ResNet 是 CV 领域最广泛验证的架构，EuroSAT 原始论文即以此为主要基准。调试资料丰富（GitHub issues/StackOverflow/官方 tutorial），降低开发风险 |
| **硬件余量充足** | 11.7M 参数，训练时（batch=256, 224×224）显存约 1.5 GB，仅为 3.0 GB 上限的 50%，为办公软件预留 > 2.5 GB |
| **训练效率极高** | 冻结骨干仅训练分类头（~5,130 参数），50 epochs 约 50 分钟，30 epochs 快速验证约 30 分钟 |
| **精度达标** | 93~95% Top-1 Accuracy 已满足需求规格的 ≥ 90% 基线。若需更高精度，解冻 layer4 微调可提升 1~3% |
| **升级路径清晰** | 冻结骨干 → 解冻 layer4 → 解冻全部 → 切换 MobileNetV3-Large，四步渐进，每步风险可控 |

**第一替代：MobileNetV3-Large（Should 优先级）**

| 选型因素 | 论证 |
|---------|------|
| **效率之王** | 参数量仅 5.4M（ResNet18 的 46%），FLOPs 仅 0.22G（12%），EuroSAT 上精度却高达 98.72% |
| **切换成本低** | TorchVision 同样提供预训练权重，代码修改仅限于 `model = torchvision.models.mobilenet_v3_large(weights='IMAGENET1K_V1')` 一行 |
| **为什么不主选** | MobileNetV3 在 EuroSAT 上的 98.7% 精度来自单一文献报告，可能存在训练协议差异导致的实际精度偏差；ResNet18 的 93~95% 来自多个文献综合估计，更稳健可预期 |

**为何不主选更高精度模型**：

| 模型 | 不主选的原因 |
|------|------------|
| ResNet50 (98.75%) | 参数 25.6M，batch=256 时显存接近 3.0 GB 上限，余量不足；精度仅比 MobileNetV3-Large 高 0.03%，效率比太低 |
| ConvNeXt-Tiny (98.74%) | 参数 28.6M，架构较新，文档和社区支持不如 ResNet；精度不优于 MobileNetV3-Large |
| ResNeXt-50 (98.96%) | 精度最高但参数 25M，性价比不如 MobileNetV3-Large |

## 6. 数据处理技术选型

### 6.1 选型总览

| 维度 | 选型 | 选型理由 |
|------|------|---------|
| 图像加载 | Pillow (PIL) + `torchvision.datasets.ImageFolder` | PyTorch 原生支持，自动按文件夹名映射类别标签 |
| 图像上采样 | `torchvision.transforms.Resize(224, BILINEAR)` | 双线性插值平滑自然，比最近邻更适合自然图像 |
| 数据标准化 | ImageNet 均值/标准差 `(0.485,0.456,0.406)/(0.229,0.224,0.225)` | 配合 ImageNet 预训练权重，标准做法 |
| 数据增强 | `torchvision.transforms` 内置操作 | 随机翻转、旋转、色彩抖动——全部在 `transforms` 包中，无需第三方增强库 |
| 数据集划分 | `sklearn.model_selection.train_test_split`（分层） | 确保各类别在训练/验证/测试集中比例一致 |
| DataLoader | `torch.utils.data.DataLoader` | 多线程加载、batch 组装、pin_memory 加速 |

### 6.2 增强库选择分析

| 候选 | 是否选择 | 理由 |
|------|---------|------|
| **torchvision.transforms** | ✅ 主选 | 内置翻转（RandomHorizontalFlip）、旋转（RandomRotation）、色彩抖动（ColorJitter），满足全部需求，零额外依赖 |
| albumentations | ❌ 不选 | 功能强大但为检测/分割任务设计，分类任务用 `transforms` 完全够用，引入 albumentations 增加依赖和维护成本 |
| timm (data) | ❌ 不选 | 增强功能与 torchvision 重叠，仅在需要 CutMix/MixUp 等高级增强时考虑（列为 Could 优先级） |

### 6.3 图像上采样方案对比

| 方案 | 描述 | 优势 | 劣势 | 选择 |
|------|------|------|------|------|
| 双线性插值 (BILINEAR) | 4 邻域加权平均 | 平滑、自然、快速 | 轻微模糊 | ✅ **主选** |
| 最近邻 (NEAREST) | 直接复制最近像素 | 速度最快 | 马赛克效应严重，不适合自然图像 | ❌ |
| 双三次插值 (BICUBIC) | 16 邻域加权 | 比双线性更平滑 | 计算量稍大，视觉差异微小 | ❌ 过度 |
| 修改 ResNet 第一层 | 改 conv1 为适配 64×64 | 保留原始信息 | 预训练权重无法利用（输入尺寸不匹配） | ❌ |

**结论**：双线性插值是最佳平衡点——计算开销小、视觉质量好、与 ImageNet 预训练权重兼容。

### 6.4 数据流设计

```

  └── SeaLake/       (2,700 张)
        │
        ▼ ImageFolder (自动按文件夹名映射 0~9 标签)
```

## 7. 训练策略技术选型

### 7.1 选型总览

| 维度 | 选型 | 参数 | 选型理由 |
|------|------|------|---------|
| 迁移学习策略 | 冻结骨干 + 训练分类头 | 可训练参数 ~5,130（0.04%） | 最大化预训练权重利用，训练极快 |
| 优化器 | AdamW | lr=1e-4, weight_decay=5e-4 | 自适应学习率 + 解耦权重衰减，迁移学习标准选择 |
| 损失函数 | CrossEntropyLoss | — | 多分类标准损失，PyTorch 内置 |
| 学习率调度 | ReduceLROnPlateau | patience=5, factor=0.5 | 监控 val_loss 自动降低学习率，简单有效 |
| 早停 | Early Stopping | patience=10, min_delta=1e-3 | 防止过拟合，自动选择最佳模型 |
| 训练精度 | FP32 | 不启用 AMP | 图像小显存充裕，FP16 无加速收益（Pascal 无 FP16 Tensor Core） |
| 批大小 | 256 | — | 最大化 GPU 利用率，显存余量充足 |

### 7.2 优化器选型分析

| 候选 | 是否选择 | 理由 |
|------|---------|------|
| **AdamW** | ✅ 主选 | 自适应学习率 + 解耦权重衰减（比 Adam 泛化更好），迁移学习分类头训练的首选。lr=1e-4 对随机初始化分类头安全 |
| Adam | ❌ 不选 | 权重衰减实现与自适应学习率耦合，泛化不如 AdamW |
| SGD + Momentum | ❌ 不选 | 需手动调参（lr schedule + warmup），对分类头训练调参成本高，收益有限 |
| RMSprop | ❌ 不选 | 主要适用于 RNN/序列任务，分类任务无明显优势 |

### 7.3 学习率调度策略分析

| 候选 | 是否选择 | 理由 |
|------|---------|------|
| **ReduceLROnPlateau** | ✅ 主选 | 监控 val_loss，停滞时自动降 lr（×0.5），简单稳健，无需预知训练时长 |
| CosineAnnealingLR | ✅ 备选 | 平滑衰减至接近 0，适合固定 epoch 数训练。配置文件中保留为可切换选项 |
| StepLR | ❌ 不选 | 需手动指定衰减节点，灵活性差 |
| OneCycleLR | ❌ 不选 | 需要从低 lr→高→低周期性调整，调参复杂，对冻结骨干的分类头训练收益有限 |
| Warmup + Cosine | ❌ 不选 | Warmup 主要解决大批量训练初期不稳定问题，本项目 batch=256 稳定，不需要 |

### 7.4 混合精度训练（AMP）决策分析

| 因素 | 分析 | 结论 |
|------|------|------|
| 显存节省 | AMP 可节省约 30% 显存 | 当前显存仅 1.5~2.0 GB，远低于 3.0 GB 上限，无需节省 |
| 训练加速 | AMP 加速依赖 FP16 Tensor Core | GTX 1050 Ti Pascal 架构无 FP16 Tensor Core，AMP **无加速效果** |
| 精度风险 | FP16 可能引入数值不稳定 | 虽然分类任务通常安全，但无收益时不应引入额外风险 |
| 代码复杂度 | AMP 需 `autocast()` + `GradScaler()` | 增加代码路径，维护成本上升 |

**决策：不启用 AMP**。在 Pascal 架构上 FP16 仅节省显存但无速度提升，而当前显存已充裕，投入产出比为负。

### 7.5 训练策略决策树

```
加载预训练 ResNet18 (ImageNet-1K)
  │
  ├─ 冻结骨干 (requires_grad=False)
  ├─ 替换 fc 层: 512 → 10
  │
  ▼
训练阶段 (50 epochs, batch=256, FP32)
  │
  ├─ 优化器: AdamW (lr=1e-4, wd=5e-4)
  ├─ 损失: CrossEntropyLoss
  ├─ 调度: ReduceLROnPlateau(patience=5, factor=0.5)
  ├─ 早停: patience=10, min_delta=1e-3
  │
  ├─ 每 epoch: 训练 → 验证 → 记录指标 → 检查早停
  ├─ 验证最佳: 覆盖保存 best_model.pth
  └─ 每 10 epoch: 保存定期检查点
  │
  ▼
精度是否 ≥ 90%?
  ├─ ✅ 是 → 完成，进入评估阶段
  └─ ❌ 否 → 升级路径:
       ├─ ① 解冻 layer4，lr=1e-5，再训练 20 epochs
       ├─ ② 切换为 MobileNetV3-Large，重新训练
       └─ ③ 启用 CutMix/MixUp 高级增强
```

| 混淆矩阵 | sklearn + seaborn 热力图 | sklearn 计算矩阵 + seaborn 渲染，组合成熟 |

| 报告格式 | Markdown + CSV | 文本报告用 Markdown（表格+分析），误分类列表用 CSV |

### 8.2 指标计算选择分析

| 方案 | 是否选择 | 理由 |
|------|---------|------|
| **手动实现 + sklearn 交叉验证** | ✅ 主选 | Accuracy/Precision/Recall/F1 公式简单（TP/FP/FN/TN 累加即可），手动实现保证透明性；sklearn 的 `classification_report` 和 `confusion_matrix` 作为交叉验证 |
| torchmetrics | ❌ 不选 | 功能强大但引入额外依赖，对 10 类分类任务过于重型 |
| sklearn 全权 | ❌ 不选 | sklearn 默认在 CPU 运行，需要将 GPU tensor 迁移，增加数据传输开销；逐 batch 累加统计量的方式更高效 |

**实现策略**：训练/验证阶段在 GPU 上逐 batch 累加 correct 计数，epoch 结束时计算 Accuracy；测试集评估阶段收集所有预测后统一计算全部指标。

## 9. 推理与部署技术选型

### 9.1 选型总览

| 维度 | 选型 | 选型理由 |
|------|------|---------|
| 推理框架 | PyTorch (eager mode) | 模型本身轻量（< 50 MB），无需导出即可高效推理 |
| 预处理管线 | 与训练完全一致（ReSize + Normalize） | 保证训练/推理数据分布一致 |
| 输出格式 | JSON + CSV | JSON 适合单张结构化输出，CSV 适合批量结果 |
| 模型导出（Could） | ONNX → ONNX Runtime | 后续优化：跨框架部署、CPU 加速 |
| 模型量化（Could） | — | 当前 Pascal 架构无 FP16 加速，量化收益有限 |

### 9.2 模型导出方案（Could 优先级）

| 格式 | 优势 | 劣势 | 当前适配 |
|------|------|------|---------|
| PyTorch (.pth) | 无需转换，直接加载 | 仅限 PyTorch 环境 | ✅ 主选 |
| ONNX (.onnx) | 跨框架，ONNX Runtime 加速 CPU 推理 | 部分算子可能不支持 | Could |
| TorchScript (.pt) | PyTorch 原生 JIT，C++ 可调用 | 对动态控制流支持有限 | Could |
| TensorRT (.engine) | NVIDIA GPU 极致加速 | Pascal 架构 FP16 无加速，导出复杂 | Won't |

**决策**：V1.0 仅使用 PyTorch `.pth` 格式进行推理，ONNX 导出列为 Could 优先级（后续如需跨平台部署或 CPU 推理加速时启用）。
